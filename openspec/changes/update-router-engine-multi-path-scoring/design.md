## Context
The current router in `/Users/rob/Dev/FerryMapper/app.js` uses a 0-1 BFS to find one abstract path that minimizes transfers first and stop hops second. Only after that path is chosen does `resolveScheduleAt()` apply GTFS schedules. This keeps the code simple, but it can choose the wrong transfer hub when a near-best topology has better timing.

## Goals / Non-Goals
- Goals:
  - Improve itinerary quality by considering multiple topology candidates before final selection
  - Preserve the no-build, client-side architecture and reuse the existing GTFS schedule resolver where practical
  - Keep response time acceptable for interactive use in the browser
  - Make route choice explainable (why this path was selected)
- Non-Goals:
  - Full replacement with RAPTOR/CSA or another full time-dependent transit engine in this change
  - Backend services or server-side routing
  - Changes to GTFS preprocessing format unless required for lightweight indexing

## Decisions

### Decision: Use bounded multi-path candidate enumeration on the existing route graph
- The router will enumerate a small set of unique candidate topology paths (for example, 5-20) instead of returning only the first 0-1 BFS winner.
- Candidate search will continue to operate on the current graph/state model (stops/routes) so the existing routing and leg reconstruction logic can be reused.
- Candidate selection will be bounded by a configurable cap and pruning rules to prevent runaway search.

Alternatives considered:
- Repeatedly rerun the current 0-1 BFS with ad hoc penalties for previous transfer stops: simpler to prototype but harder to reason about completeness and duplicate paths.
- Full time-dependent routing engine (RAPTOR/CSA): likely best long-term, but too large for an initial improvement and higher risk in a no-build client app.

### Decision: Score candidate paths using resolved schedule outcomes
- For each candidate topology, the existing schedule resolution flow will compute real departures/arrivals and transfer waits.
- The final selected itinerary will be chosen using mode-aware ranking on resolved times, not topology cost.
- Topology metrics (transfers, stops) remain tie-breakers, not primary selection criteria.

Alternatives considered:
- Continue topology-first selection and only adjust leg route IDs opportunistically (`findNextTripAnyRoute`): insufficient because the transfer hub/path is already fixed.

### Decision: Apply candidate scoring in option generation (`Earlier / Requested / Later`)
- Each time window candidate generated by `findOptions()` will evaluate the candidate path set and keep the best resolved itinerary for that time.
- This prevents the requested tab from improving while `Earlier/Later` remain locked to an inferior topology.

Alternatives considered:
- Only score multiple paths for the `Requested` tab: lower implementation cost, but inconsistent option quality.

### Decision: Return router decision metadata for explainability
- The routing result will include structured metadata summarizing:
  - chosen topology/transfer stops
  - runner-up topology summaries
  - ranking reason (for example, earlier arrival or later departure while still arriving by target)
- UI display is optional, but metadata will be available for debugging and future UX improvements.

Alternatives considered:
- No metadata: simpler, but makes route-choice changes harder to trust/debug.

## Risks / Trade-offs
- More candidate evaluation increases CPU time in the browser.
  - Mitigation: strict candidate caps, deduplication, and schedule lookup indexing/caching.
- More complex ranking can create unexpected changes in route choice.
  - Mitigation: explicit ranking order in spec + reason metadata in UI/debug output.
- `Arrive by` logic still uses window scanning in this change.
  - Mitigation: improve selection quality within current scanning approach now; defer exact backward search to a future change.

## Migration Plan
1. Implement candidate-path enumeration behind a small configurable cap.
2. Refactor schedule scoring to accept multiple candidate topologies.
3. Update option generation to score candidates per option.
4. Add decision metadata and optional UI rendering.
5. Run manual route comparisons and tune caps/tie-breakers.

## Open Questions
- What default candidate cap gives the best quality/performance trade-off on low-end devices (5, 10, 12, or 20)?
- Should the UI show one-line route-choice reasoning by default, or keep it debug-only initially?
- Should we preserve the current `0-1 BFS` output as an explicit fallback option for easy A/B testing?
